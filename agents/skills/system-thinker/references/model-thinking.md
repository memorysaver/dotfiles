# Many-Model Thinking

Use this reference when the user asks for Scott E. Page-style many-model thinking, model portfolios, or a cross-check of a systems-thinking diagram. This is a practical synthesis inspired by the many-model paradigm, not a reproduction of *The Model Thinker*.

Sources for orientation: Scott E. Page describes the book as promoting many-model thinking, "the application of ensembles of models to make sense of complex phenomena"; public catalog records list the model families covered in the book. Use `model-thinker-32-models.md` for the local model index.

## Relationship to systems thinking

Systems thinking asks:

- What behavior over time are we trying to explain?
- What feedback loops, delays, stocks, flows, and constraints produce it?
- Where can intervention change the structure?

Many-model thinking asks:

- What other simplified models could explain part of the same problem?
- What does each model make visible?
- What does each model hide?
- Which conclusions survive across several models?

Use systems thinking as the structural map. Use many-model thinking as a model portfolio around that map.

## Model portfolio workflow

1. Start with the problem behavior:
   - Name the outcome variable.
   - State the time horizon.
   - Describe the observed or hypothesized behavior-over-time pattern.

2. Build or summarize the systems model:
   - Identify the dominant feedback loops.
   - Mark delays and accumulations.
   - State the core intervention hypothesis.

3. Select 2-5 additional models:
   - Choose models because they expose different mechanisms, not because they sound impressive.
   - Include at least one model that could challenge the systems diagram.
   - Prefer models that match the available evidence.

4. Apply each model with the same standard:
   - Purpose
   - Best-fit conditions
   - Key variables
   - Mechanism
   - What it explains
   - What it hides
   - Evidence needed
   - Action implication

5. Synthesize:
   - Agreements across models.
   - Tensions across models.
   - Robust actions.
   - Uncertainty and next evidence to collect.

## Standard model output format

Use this format for every model in a portfolio:

```text
Model: [model name]
Use it when: [conditions where this model fits]
Core question: [the question this model answers]
Key variables: [3-6 variables]
Mechanism: [one-paragraph explanation]
What it explains: [behavior/pattern it can explain]
What it hides: [blind spots and simplifications]
Evidence to check: [data/observations that would support or weaken it]
Action implication: [what this model suggests doing]
How to read the result: [plain-language reading guide]
```

## Model reading standard

When teaching a user how to read model-thinking output, use this sequence:

1. Identify the model's purpose:
   - Is it for explanation, prediction, design, communication, action, or exploration?

2. Identify the simplification:
   - What did the model leave out so it could become useful?

3. Identify the mechanism:
   - What process produces the result?

4. Identify the output:
   - What pattern, ranking, probability, equilibrium, threshold, trajectory, or policy implication does it produce?

5. Identify the failure mode:
   - When would this model mislead us?

6. Compare with other models:
   - Does this model confirm, challenge, or refine the systems diagram?

## Portfolio synthesis format

After applying several models, summarize with:

```text
Convergent insight:
[What several models agree on.]

Tension:
[Where models disagree or emphasize different risks.]

Robust action:
[Action that still looks good under several models.]

Fragile action:
[Action that depends heavily on one model being true.]

Next evidence:
[Observation, data, or experiment that would improve the model portfolio.]
```

## Quality checks

- Do not apply many models as a checklist. Select models that fit the question.
- Do not treat a model portfolio as proof. It is a structured way to reduce blind spots.
- Do not hide assumptions. A model's usefulness depends on when its assumptions are approximately true.
- Prefer diverse mechanisms: feedback, incentives, networks, uncertainty, learning, thresholds, and path dependence.
- Keep the user's cognitive load low: explain 2-5 models well before listing more.
